# Report Rebuild Tasks

目標：重新抓取並整理 oloo 實驗資料，重算報告所需統計，修正報告內容邏輯，最後檢查 A4 版面與 PDF 匯出流程。

## 進度

- [x] 1. 盤點目前報告內容與使用者 20 點修正意見，標記已符合、待重算、待改寫項目。
- [x] 2. 盤點遠端 `emfox` 的資料來源：PostgreSQL / TimescaleDB、既有 CSV、既有 dump、資料時間範圍、欄位可用性。
- [x] 3. 建立可重跑的資料匯出程式，放在 repo 內，從 `ssh emfox` 抓取最新統計結果。
- [x] 4. 重算站點層級資料：每日、星期一、星期五、校區分開、每小時可借車數、總車數、缺車率、低庫存率。
- [x] 5. 重算站點缺車拆解：完全沒車 vs 有車但不可借，並產生 Top 10 圖表/表格資料。
- [x] 6. 重算車況原因：低電量區間、官方 `error_msg` 類型、不可借原因並列比較。
- [x] 7. 重算補車/恢復分析：缺車 episode、多久恢復、疑似人工補車 vs 一般歸還或零散恢復。
- [x] 8. 檢查路線分析可行性：若站點轉換資料不足，明確寫成限制；若足夠，產生路線後不可借或故障分析。
- [x] 9. 產出報告資料檔並改成由資料檔驅動，避免手動硬寫統計數字。
- [x] 10. 改寫報告主文：回答「為什麼借不到車」與「怎麼比較借得到車」，並整理摘要、方法、限制、結論脈絡。
- [x] 11. 修正所有已知文字問題：`oloo` 大小寫、標題、觀測窗、低庫存定義、補車定義、header/footer、頁碼。
- [x] 12. 檢查外觀：A4 每頁排版、邊界、頁碼、圖表 caption、表格可讀性、避免大片空白。
- [x] 13. 執行驗證：format、lint、typecheck、build、報告 dev server HTTP 檢查。
- [x] 14. 最終人工閱讀檢查：逐頁確認邏輯順、問題有回答、排版舒適。
